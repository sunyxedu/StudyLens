# StudyLens Architecture

StudyLens is split into four bounded areas:

1. Ingestion adapters collect Imperial course artifacts. Scientia, Panopto, and EdStem all sit behind Imperial SSO; all three share a single `BrowserSession` (a Playwright `BrowserContext` loaded from saved storage state). The web API loads that state from the current user's encrypted SQLite record, while the CLI can still load a legacy `BROWSER_STORAGE_STATE` file. Static HTML / PDF fetches go through the context's `request` API; DOM-driven flows open a `Page` from the same context so cookies and redirects stay consistent. The Scientia timeline page itself is parsed by an LLM extractor (`LLMCourseExtractor`, default `claude-sonnet-4-6`) with prompt caching, so structural changes to Imperial's HTML don't break course discovery. The auto-index pipeline discovers Scientia courses, downloads supported materials/exercises/tutorials, extracts text, and indexes chunks. Panopto discovery is driven by a Claude Agent SDK loop (`panopto_agent.py`): a small set of Playwright-backed tools (`goto`, `page_summary`, `list_links`, `click_text`, `search_panopto`, terminal `submit_videos`) lets the model open Shared-with-Me, locate the course's official folder, and list its videos. Video bytes are then handled deterministically — caption/subtitle indexing with timestamps, falling back to Whisper transcription when captions are unavailable and a downloadable video plus OpenAI key are available, pulls past exam papers from `exams.doc.ic.ac.uk` over HTTP Basic auth (using `IMPERIAL_USERNAME` / `IMPERIAL_PASSWORD`), and scrapes scope-relevant EdStem posts as `edstem_note` chunks. Each ingestion stage skips cleanly when its credentials aren't configured.
2. Document processing extracts text and creates stable `DocumentChunk` objects.
3. Retrieval embeds chunks and stores them in Qdrant by default, with course/resource metadata as payload filters. Search is adaptive: it fetches the requested `top_k` (default 5), collapses duplicate texts, then a relevance judge grades each distinct chunk and the window doubles (5 -> 10 -> 20 -> 40 ...) while more than half of each newly fetched batch is judged relevant, capped at `ADAPTIVE_MAX_K` (default 80). The judge is a dedicated temperature-0 client on `OPENAI_JUDGE_MODEL` (default `gpt-4.1`, stronger than the answer model because its verdicts gate citations; keyword overlap offline). Only judge-approved chunks reach the answer prompt and the citation list — when nothing is judged relevant the answer reports missing evidence instead of citing unrelated context. `ADAPTIVE_RETRIEVAL=false` restores plain top-k, and `POST /admin/dedupe-index` collapses duplicate texts already in the store.
4. Generation uses retrieved context to answer questions, create two-page LaTeX cheatsheets, and draft predicted exam papers. Cheatsheet and predicted-paper generators automatically pull indexed `edstem_note` chunks for the course as exam-scope guidance; an explicit `scope_notes` argument overrides that auto-pull.
5. Authentication uses HttpOnly session cookies. StudyLens account passwords are one-way PBKDF2 hashes, course caches are scoped by `user_id`, and browser storage state is encrypted at rest with `AUTH_SECRET_KEY` (or a generated local secret).
6. The web UI in `web/` provides the main local workspace; the browser extension stays focused on page-level Q&A.

The ingestion pipeline is async end-to-end; the API and CLI cross the sync/async boundary with FastAPI async handlers and `asyncio.run` respectively. The browser extension never scrapes or stores credentials; it sends page context and questions to the backend API.
